Body sections:

1. **## When this skill applies** — Creating invoices, checking payment status, managing overdue invoices, generating invoices from time entries or proposals.

2. **## Tool workflow** — Numbered steps:
   1. `get_project` — confirm project and client context
   2. `list_proposals` — find accepted proposal for pricing reference (per INV-03)
   3. `aggregate_time` — if billing by hours, get total time for the billing period
   4. `create_invoice` — generate with line items, due date, and amount
   5. `get_invoice` — retrieve invoice details for review or status checking
   6. `update_invoice` — change status: draft -> sent -> paid (or overdue)
   7. If overdue: reference the freelance-followups skill for follow-up coaching

3. **## Invoice coaching principles** (per D-04, keep concise — this is the lower-priority skill):
   - **Line items over lump sums**: Always itemize work. "Website design - $3,000" is worse than "Homepage design - $1,200 / About page - $600 / Contact form - $400 / Responsive QA - $800". Itemization justifies the total and makes partial disputes easier to resolve.
   - **Reference the proposal**: Every invoice should trace back to the accepted proposal amount. If the invoice differs from the proposal (scope changes, additional work), note why.
   - **Due dates**: Default Net 30 for established clients, Net 15 for new clients. Always set an explicit due date, never "upon receipt" (too vague, no leverage).
   - **Status discipline**: Move invoices through statuses promptly. Draft -> Sent (day of delivery) -> Paid (day payment clears). Mark overdue the day after due date — don't let it slide.
   - **Invoice numbering**: Use sequential numbering (INV-001, INV-002). Never reuse numbers. This is both professional and legally required in many jurisdictions.

4. **## Standard line item structures** (per D-05):
   | Project type | Line items |
   |-------------|------------|
   | Fixed-price project | Milestone deliverables from proposal |
   | Hourly project | Time period + hours + rate per `aggregate_time` |
   | Retainer | Monthly retainer fee + overage hours if applicable |
   | Scope change add-on | Original scope items + "Additional: [change]" line |

5. **## Conditional coaching**:
   - If creating from scratch: walk through line items, reference proposal, set due date
   - If user has details ready: create efficiently, offer quick review
   - If checking status: use `list_invoices` with status filter, flag any overdue

- Include trigger phrases: "log time", "track hours", "how many hours", "time for [project]", "billable hours", "time report", "aggregate time"
- Under 1,024 characters

Body sections:

1. **## When this skill applies** — Logging time entries, reviewing hours per project, aggregating time for invoicing, time-based billing decisions.

2. **## Tool workflow** — Numbered steps:
   1. `get_project` — confirm which project to log time against
   2. `create_time_entry` — log the entry with description, duration, and date
   3. `list_time_entries` — review entries for a project or date range
   4. `aggregate_time` — total hours for invoicing. Call this before `create_invoice` to get accurate billable hours.
   5. For invoicing: hand off to freelance-invoices skill

3. **## Time tracking principles** (per D-04, concise):
   - **Log daily, not weekly**: Reconstruct time from memory and you'll undercount by 15-25%. Log at the end of each work session.
   - **Descriptive entries**: "Website work - 3 hours" tells the client nothing. "Homepage hero section design and responsive testing - 3 hours" justifies the bill. Clients dispute vague entries.
   - **Round to nearest 15 minutes**: Industry standard. 0.25 hour increments. Don't bill for 2 minutes of email; batch small tasks.
   - **Separate billable and non-billable**: Admin, internal meetings, and rework from your own errors are non-billable. Only bill for client-facing value.
   - **Track before aggregating**: Always run `list_time_entries` before `aggregate_time` to spot missing entries or duplicates. Aggregate is only as good as its inputs.

4. **## Conditional coaching**:
   - If logging time: remind about descriptive entries if description is vague
   - If aggregating for invoice: prompt to review entries first, then hand off to invoice workflow
   - If user just wants to log quickly: persist immediately, no coaching
