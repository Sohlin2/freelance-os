**Proposals SKILL.md** (~2,200 tokens / ~8,800 chars target per D-10):

Frontmatter (YAML between `---` delimiters):
```yaml
name: freelance-proposals
description: >
  Expert coaching for freelance project proposals. Guides pricing,
  scope clarity, revision limits, and payment terms. Use when drafting,
  reviewing, pricing, or sending a proposal for a client project.
```
- Description must be under 1,024 characters and include trigger phrases: "draft a proposal", "write a proposal", "price this project", "proposal for [client]", "what should I include in my proposal", "review my proposal"
- Do NOT include `disable-model-invocation` (must auto-invoke per D-07/SKLL-02)

Body sections (use markdown headers, tables, numbered lists — scannable, not paragraphs per research anti-patterns):

1. **## When this skill applies** — Brief 2-3 line description of when Claude should activate this skill. Include: drafting new proposals, reviewing existing proposals, pricing discussions, scope negotiation.

2. **## Tool workflow** — Numbered steps per D-08:
   1. `list_clients` or `get_client` — confirm client context (billing rate, past projects)
   2. `list_proposals` — check prior proposals for this client (avoid duplicate pricing)
   3. `get_project` — confirm project details and status
   4. Draft proposal content applying coaching principles below
   5. `create_proposal` — persist with status: 'draft'
   6. Remind user to `update_proposal` with status: 'sent' after delivery

3. **## Coaching principles** (per D-04, prescriptive with specific defaults):
   - **Pricing**: Always calculate from time estimate x rate, not gut feeling. Ask "how many hours?" before pricing. If client has a billing rate on file, use it. For fixed-price, add 20% buffer for unknowns. Never price below $50/hour for professional services.
   - **Scope clarity**: Every proposal must define what IS included and what is NOT included (exclusions section). Vague scope = scope creep.
   - **Revision rounds**: Include 2-3 revision rounds in the proposal. State explicitly: "Additional revisions at $X/hour." This is the #1 source of scope creep for freelancers.
   - **Payment terms**: Default to 50% upfront, 50% on delivery for projects over $2,000. For smaller projects, 100% on completion with Net 15 terms. For ongoing work, Net 30 with monthly invoicing. Always specify late payment fee (1.5%/month industry standard).
   - **Validity**: Proposals expire in 30 days. State this explicitly. Prevents "I want to go ahead with that proposal from 6 months ago" at outdated rates.
   - **Deliverables**: List specific deliverables, not vague outcomes. "5-page marketing website with responsive design" not "a website".

4. **## Proposal structure outline** (per D-05, structural outline not full draft to save tokens):
   Table format:
   | Section | What to include |
   |---------|----------------|
   | Project summary | 2-3 sentences: client name, project goal, timeline |
   | Scope of work | Numbered deliverables list |
   | Exclusions | What is NOT included (critical for scope protection) |
   | Timeline | Milestones with dates or week numbers |
   | Pricing | Line items or fixed price with breakdown |
   | Payment terms | Schedule, method, late fee |
   | Revisions | Number included, hourly rate for extras |
   | Validity | Expiration date (30 days from creation) |
   | Next steps | How to accept (sign, reply, deposit) |

5. **## Conditional coaching** (per research Pitfall 4):
   - If user is drafting from scratch: apply full coaching, ask clarifying questions about scope/timeline/rate before drafting
   - If user has already written content and wants to store it: offer a quick review ("Want me to check pricing and scope clarity before saving?") instead of full coaching lecture
   - If user says "just save it": persist immediately with `create_proposal`, no coaching

1. **## When this skill applies** — Drafting follow-up emails, planning follow-up timing, choosing tone for client communications, managing overdue invoices or unanswered proposals.

2. **## Tool workflow** — Numbered steps per D-08:
   1. `get_followup_context` — ALWAYS call this first. It returns client info, project details, recent follow-up history, and invoice status. This is the data foundation for the email.
   2. Analyze context: determine follow-up type, days since last contact, invoice status
   3. Apply timing/tone matrix below
   4. Draft the follow-up content
   5. `create_followup` — persist with type field matching context
   6. After user confirms sending: `mark_followup_sent`

3. **## Timing and tone matrix** (per FLLW-02, research Pattern 4 — this is the core coaching table):

   | Context | `type` field value | When to send | Tone | Opening line example |
   |---------|-------------------|-------------|------|---------------------|
   | Awaiting proposal response | proposal_follow_up | 3 business days after sending | Warm, brief | "Just checking in on the proposal I sent over for [project]." |
   | Invoice approaching due | invoice_reminder | 1 business day before due date | Professional, helpful | "Friendly reminder that invoice #[number] for $[amount] is due [date]." |
   | Invoice overdue (1-7 days) | invoice_overdue | 3 days after due date | Firm but friendly | "Following up on invoice #[number] — it was due on [date]. Let me know if you need anything from my end." |
   | Invoice overdue (7+ days) | invoice_overdue | 7 days, then 14 days | Direct, professional | "Invoice #[number] is now [X] days past due. Please confirm payment timeline." |
   | General check-in | check_in | 4-6 weeks of no contact | Relationship-first, no asks | "It's been a while — wanted to see how [project/business] is going." |
   | Awaiting client feedback | awaiting_response | 48 hours after request | Light touch | "Did you get a chance to review [deliverable]? Happy to walk through it." |
   | Project completion follow-up | check_in | 2 weeks after delivery | Grateful, future-oriented | "Hope [deliverable] is working well. Would love to hear how it's going." |

4. **## Follow-up principles**:
   - Never send more than 3 follow-ups for the same item without a response. After 3, wait for client to re-engage or have a direct conversation.
   - Always reference specific data: invoice number, amount, project name, date sent. Generic follow-ups feel automated.
   - Check `list_followups` before drafting — avoid sending duplicate follow-ups on the same day or for the same item.
   - Morning emails (9-10 AM client's timezone) get higher response rates than afternoon.
   - Shorter is better. Follow-ups should be 3-5 sentences max. One clear ask per email.

5. **## Conditional coaching** (per Pitfall 4):
   - If user is asking Claude to draft: apply full timing/tone coaching
   - If user has written content and wants to save: offer quick tone check ("This reads a bit strong for a first reminder — want me to soften it?")
   - If user says "just save/send it": persist immediately, no coaching

**Scope SKILL.md** (~1,500 tokens / ~6,000 chars target per D-10):

Frontmatter:
```yaml
name: freelance-scope
description: >
  Expert coaching for freelance project scope management. Guides scope
  definition, change request handling, and scope creep detection. Use
  when defining project scope, logging change requests, or reviewing
  scope boundaries.
```
- Include trigger phrases: "define scope", "scope for this project", "client wants changes", "is this in scope", "scope creep", "change request", "out of scope"
- Under 1,024 characters

Body sections:

1. **## When this skill applies** — Defining initial project scope, evaluating change requests, detecting scope creep, reviewing scope boundaries, and managing client expectations around deliverables.

2. **## Tool workflow** — Two workflows:

   **Defining scope (new project):**
   1. `get_project` — confirm project exists and is active
   2. `list_proposals` — check if accepted proposal exists (scope may already be seeded by `accept_proposal`)
   3. `get_scope` — check if scope definition already exists
   4. If no scope: `create_scope` with deliverables, boundaries, exclusions
   5. If scope exists (seeded from proposal): `update_scope` to refine boundaries/exclusions

   **Handling change requests:**
   1. `get_scope` — load current scope definition
   2. `check_scope` — evaluate whether the new request falls within or outside agreed scope. This tool returns the scope definition AND recent scope changes for context.
   3. If in-scope: `log_scope_change` with classification `in_scope`
   4. If out-of-scope: coach user on how to communicate the change to the client (see below), then `log_scope_change` with classification `out_of_scope`
   5. `list_scope_changes` — show scope change history to illustrate pattern

3. **## Scope coaching principles** (per D-04):
   - **Define boundaries early**: Every scope definition must include an "Exclusions" section. What you DON'T include is more important than what you do. Clients remember "you said you'd handle the website" but forget "hosting and domain registration were explicitly excluded."
   - **The 3-strike rule for creep**: If a client makes 3+ out-of-scope requests, it's time for a scope conversation, not another accommodation. Use `list_scope_changes` to show the pattern.
   - **Change request response template**: "Happy to take that on! Since [request] falls outside our agreed scope, I'll send over a quick addendum with the additional cost and timeline. Does that work?" — Always frame as collaborative, not adversarial.
   - **Scope vs. revision**: Revisions = changes to agreed deliverables (covered by revision rounds in proposal). Scope changes = NEW deliverables not in the original agreement. Teach the client this distinction early.
   - **Budget impact**: Every out-of-scope item should have an estimated cost. Never say "this is out of scope" without immediately offering a path forward with pricing.

4. **## Scope definition checklist**:
   | Element | Required | Example |
   |---------|----------|---------|
   | Deliverables | Yes | "5-page responsive website, logo design, content writing" |
   | Boundaries | Yes | "Up to 5 pages, 2 revision rounds, desktop + mobile" |
   | Exclusions | Yes | "Hosting, domain, ongoing maintenance, stock photos" |
   | Timeline | Yes | "4 weeks from project start" |
   | Change process | Recommended | "Changes evaluated within 48h, priced if out-of-scope" |

5. **## Conditional coaching**:
   - If user is defining scope from scratch: apply full coaching, ensure exclusions are defined
   - If evaluating a change request: focus on in-scope vs out-of-scope classification and response coaching
   - If user just wants to log something: persist immediately
