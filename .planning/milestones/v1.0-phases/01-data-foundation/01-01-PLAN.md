1. Create migration 20260328000001_create_extensions.sql:
```sql
-- Enable required extensions in the extensions schema
create extension if not exists pgcrypto schema extensions;
create extension if not exists moddatetime schema extensions;
```

IMPORTANT: Use `schema extensions` — not bare `create extension` — because Supabase migrations run with a search_path that does not include the extensions schema by default (per Pitfall 2 and 3 from research).

2. Create migration 20260328000002_create_enums_and_helpers.sql:
```sql
-- Enum types (per D-02, values from research Domain Table Schema section)
create type project_status as enum ('active', 'paused', 'completed');
create type proposal_status as enum ('draft', 'sent', 'accepted', 'declined', 'expired');
create type invoice_status as enum ('draft', 'sent', 'paid', 'overdue', 'void');
create type scope_change_classification as enum ('in_scope', 'out_of_scope', 'needs_review');
create type follow_up_type as enum ('proposal_follow_up', 'invoice_overdue', 'check_in', 'awaiting_response', 'other');

-- Helper function for RLS policies (per D-09, with Pitfall 4 fix: NULLIF for empty string)
create or replace function public.current_app_user_id()
returns uuid
language sql stable
as $$
  select nullif(current_setting('app.current_user_id', true), '')::uuid;
$$;

comment on function public.current_app_user_id() is
  'Returns the current authenticated user ID from the session variable set by the MCP server. Returns NULL if not set.';
```

The NULLIF handles the case where current_setting returns empty string instead of NULL when the variable is not set (Pitfall 4). Without it, casting empty string to UUID throws an error.

The function is marked STABLE (not VOLATILE) because it reads a session variable that does not change within a single transaction.
  </action>
