# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Full Tool Suite.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> NOTE: FreelanceOS has no visual frontend. The "UI" for this product is the
> conversational interface Claude presents to the freelancer: tool description
> strings (which Claude uses to decide when to call a tool), JSON response
> formatting (what the freelancer reads back), and error copy. This spec
> governs those text contracts. All six sections are adapted for a
> conversational/MCP context rather than a graphical one.

Not applicable to this phase. The output surface is Claude Code's chat terminal, not a rendered DOM. There are no layout elements, padding values, or margin tokens to specify.

## Copywriting Contract

This is the primary applicable section for a conversational MCP product. All copy governs what Claude reads (tool descriptions) and what the freelancer reads (response text).

### Tool Description Pattern

**Format:** `{Action stated as verb phrase}. Use when {trigger condition}.`

**Character budget:** 120 characters maximum per tool description. Claude truncates long descriptions in tool manifests.

**Requirement:** SKLL-03 mandates total tool manifest stays under 15,000 tokens. Concise descriptions are a hard constraint.

| Tool | Description |
|------|-------------|
| `create_proposal` | Store a new proposal for a project. Use when the freelancer has drafted proposal content and wants to save it. |
| `get_proposal` | Retrieve a single proposal by ID. Use when the freelancer asks to view or reference a specific proposal. |
| `list_proposals` | List proposals for a project or client. Use when the freelancer asks for proposal history or wants to check existing proposals. |
| `update_proposal` | Update proposal fields or status. Use when the freelancer wants to change proposal content, status, or details. |
| `accept_proposal` | Mark a proposal as accepted and seed the project scope from its deliverables. Use when the freelancer confirms a proposal was accepted by the client. |
| `create_invoice` | Store a new invoice with line items and totals. Use when the freelancer wants to generate and save an invoice for a project. |
| `get_invoice` | Retrieve a single invoice by ID. Use when the freelancer asks to view a specific invoice. |
| `list_invoices` | List invoices filtered by status, client, or date range. Use when the freelancer asks about outstanding, paid, or overdue invoices. |
| `update_invoice` | Update invoice fields or status. Use when the freelancer marks an invoice as sent, paid, or overdue. |
| `create_time_entry` | Log a time entry against a project. Use when the freelancer reports hours worked on a project. |
| `get_time_entry` | Retrieve a single time entry by ID. Use when the freelancer asks to view a specific logged time entry. |
| `list_time_entries` | List time entries for a project, optionally filtered by date range. Use when reviewing logged hours for a project. |
| `update_time_entry` | Update a time entry's duration, date, or description. Use when the freelancer wants to correct a logged entry. |
| `delete_time_entry` | Delete a time entry permanently. Use when the freelancer wants to remove an incorrectly logged entry. |
| `aggregate_time` | Sum total hours and amount for a project over an optional date range. Use when preparing an invoice or checking billable hours. |
| `create_scope` | Define the agreed project scope with deliverables and boundaries. Use when starting a project or after a proposal is accepted. |
| `get_scope` | Retrieve the agreed scope for a project. Use when the freelancer wants to review what was agreed. |
| `list_scope_changes` | List scope change requests logged against a project. Use when reviewing scope creep history or change log. |
| `log_scope_change` | Record a scope change request with classification. Use ONLY after the freelancer confirms they want to log the change. |
| `check_scope` | Retrieve agreed scope and change history so Claude can assess whether a new request falls within scope. Use when a client requests something new and the freelancer wants to check if it is in scope. |
| `create_followup` | Store a drafted follow-up message. Use when the freelancer has composed a follow-up and wants to save it. |
| `get_followup` | Retrieve a single follow-up by ID. Use when the freelancer asks to view a specific follow-up record. |
| `list_followups` | List follow-ups for a client or project. Use when the freelancer asks to review follow-up history. |
| `mark_followup_sent` | Mark a follow-up as sent. Use when the freelancer confirms they have sent the follow-up to the client. |
| `get_followup_context` | Fetch all relevant context for drafting a follow-up: overdue invoices, project status, days since last contact, prior follow-ups. Use before drafting a follow-up message. |

### Error State Copy

**Pattern:** `"Failed to {operation}: {db_error_message}"`

All error messages follow the Phase 2 established pattern. No custom error prose — pass the database error message through directly so the freelancer gets the real reason.

| Scenario | Copy |
|----------|------|
| DB insert failure | `"Failed to create {entity}: {error.message}"` |
| DB read failure | `"Failed to retrieve {entity}: {error.message}"` |
| DB update failure | `"Failed to update {entity}: {error.message}"` |
| DB delete failure | `"Failed to delete {entity}: {error.message}"` |
| Record not found | `"Failed to retrieve {entity}: {error.message}"` (Supabase returns PGRST116 for single() with no rows) |
| Unexpected thrown error | `"Failed to {operation}: {err.message}"` via `catch (err: unknown)` block |

**Source:** Phase 2 pattern in `src/tools/clients.ts` — carry forward unchanged.

### Empty State Copy

Empty states surface as JSON responses. When a list tool returns zero results, the response is `[]`. Claude interprets this and informs the freelancer conversationally. No custom empty-state messages required inside tool handlers.

**Exception — `check_scope` with no scope defined:** If `get_scope` returns null for the project, the tool response must include an explicit signal so Claude does not misread it as "scope is unconstrained." Response must be:

```json
{
  "scope": null,
  "scope_changes": [],
  "message": "No scope has been defined for this project yet."
}
```

**Exception — `get_followup_context` with no overdue invoices:** Response must be a complete object even when sub-arrays are empty, so Claude has a full picture:

```json
{
  "overdue_invoices": [],
  "project_status": "active",
  "days_since_last_contact": null,
  "prior_followups": []
}
```

### Destructive Action Copy

Two tools in this phase perform irreversible or high-consequence operations. Claude MUST ask the freelancer to confirm before calling these tools.

| Tool | Consequence | Confirmation requirement |
|------|------------|--------------------------|
| `delete_time_entry` | Permanent deletion — no soft delete, no recovery | Claude asks: "Permanently delete this time entry for {duration} on {date}? This cannot be undone." Wait for explicit confirmation before calling. |
| `log_scope_change` | Writes a formal scope change to the record — affects client relationship history | Per D-14 (CONTEXT.md): Claude always asks the user before logging a scope change. Claude must present classification options (in_scope / out_of_scope / needs_review) and await freelancer approval before calling. |

**Note on `accept_proposal`:** This is a consequential action (sets status to `accepted` + seeds scope in one transaction) but is not destructive. It is reversible only by directly updating both records. Claude should confirm: "Accept this proposal and create the project scope from its deliverables?" before calling.

### Primary CTA Labels

There is no visual CTA in this product. The conversational equivalent — the action phrase Claude uses when prompting the freelancer — follows this pattern:

| Action | Claude prompt phrasing |
|--------|------------------------|
| Create proposal | "Save this proposal to the project?" |
| Accept proposal | "Mark this proposal as accepted and set up the project scope?" |
| Generate invoice | "Create this invoice for {client}?" |
| Log time | "Log {N} hours to {project}?" |
| Mark invoice paid | "Mark invoice #{id} as paid?" |
| Log scope change | "Log this as a {classification} scope change?" |
| Send follow-up | "Mark this follow-up as sent?" |
