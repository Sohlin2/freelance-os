**Tool 1: create_proposal** (per D-01, D-05, PROP-01)
- Description: "Store a new proposal for a project. Use when the freelancer has drafted proposal content and wants to save it."
- inputSchema: client_id (uuid required), project_id (uuid required), title (string min 1), content (string optional — free-form text), amount (number positive optional), currency (string length 3 default 'USD'), valid_until (string date optional)
- Insert into `proposals` table with `{ ...args, user_id: userId }`, `.select().single()`
- Error: "Failed to create proposal: {error.message}"

**Tool 2: get_proposal** (per D-05, PROP-04)
- Description: "Retrieve a single proposal by ID. Use when the freelancer asks to view or reference a specific proposal."
- inputSchema: proposal_id (uuid required)
- Select from `proposals` where id = proposal_id AND archived_at IS NULL, `.single()`
- Not found: "Proposal not found or has been archived"

**Tool 3: list_proposals** (per D-05, PROP-04)
- Description: "List proposals for a project or client. Use when the freelancer asks for proposal history or wants to check existing proposals."
- inputSchema: project_id (uuid optional), client_id (uuid optional), status (enum ['draft','sent','accepted','declined','expired'] optional), sort_by (enum ['created_at','updated_at','amount','title'] default 'created_at'), sort_dir (enum ['asc','desc'] default 'desc'), limit (int 1-100 default 20), offset (int min 0 default 0)
- Conditional filters: if project_id, `.eq('project_id', project_id)`; if client_id, `.eq('client_id', client_id)`; if status, `.eq('status', status)`
- Always `.is('archived_at', null)`
- Return `{ proposals: data, total: count, limit, offset }`

**Tool 4: update_proposal** (per D-05)
- Description: "Update proposal fields or status. Use when the freelancer wants to change proposal content, status, or details."
- inputSchema: proposal_id (uuid required), title (string optional), content (string optional nullable), status (enum ['draft','sent','accepted','declined','expired'] optional), amount (number positive optional nullable), currency (string length 3 optional), valid_until (string date optional nullable), sent_at (string datetime optional nullable), responded_at (string datetime optional nullable)
- Extract proposal_id, filter undefined fields, update `.eq('id', proposal_id).select().single()`
- Empty updates: "No fields to update"

**Tool 5: accept_proposal** (per D-04, PROP-03 — CRITICAL transactional tool)
- Description: "Mark a proposal as accepted and seed the project scope from its deliverables. Use when the freelancer confirms a proposal was accepted by the client."
- inputSchema: proposal_id (uuid required)
- Inside one `withUserContext` call, three sequential operations:
  1. Fetch proposal: `db.from('proposals').select('*').eq('id', proposal_id).is('archived_at', null).single()`
  2. Update status: `db.from('proposals').update({ status: 'accepted', responded_at: new Date().toISOString() }).eq('id', proposal_id).select().single()`
  3. Upsert scope: `db.from('scope_definitions').upsert({ project_id: proposal.project_id, user_id: userId, deliverables: proposal.content ?? '' }, { onConflict: 'project_id' }).select().single()`
- Return `{ proposal: updatedProposal, scope: scopeData }`
- If proposal not found at step 1, return isError with "Proposal not found"
- If update fails at step 2, return isError
- If scope upsert fails at step 3, return `{ proposal: updatedProposal, scope: null, error: scopeError.message }`

IMPORTANT: Use ACTUAL enum values from database.ts — `'declined'` not `'rejected'`, `'expired'` is valid. Do NOT use CONTEXT.md D-03 labels which are incorrect.
